# 网络请求模块详细任务清单

本文档用于指导 AI 在任意新项目结构下重建一个与当前项目网络请求模块能力和协议严谨度完全一致的实现。本文档故意不指定任何文件名，允许 AI 自由命名，但不允许降低协议一致性、错误处理完整性或调用约束。

## 1. 总目标

- 搭建一个统一的网络请求模块，覆盖普通查询、表单提交、文件上传、统一响应解析、统一错误处理、登录失效处理、动态域名初始化、公参注入、签名生成、加密上报。
- 输出的能力必须与现有模块保持行为一致，而不只是接口名相似。
- 可变信息如域名、公参映射、签名密钥、加密密钥、远端配置地址可以留空或抽成占位配置，但调用链路和处理顺序必须保留。

## 2. 架构拆分任务

- 定义一个“请求客户端”职责层，统一持有底层 HTTP 实例。
- 定义一个“请求配置管理”职责层，负责基础域名、公参来源、公参映射、签名拼装、随机串生成、远端域名初始化。
- 定义一个“响应数据处理”职责层，负责把任意原始响应转换成统一响应壳对象。
- 定义一个“统一响应模型”职责层，至少包含业务状态码、业务提示文案、业务数据载荷三个核心字段。
- 定义一个“异常与错误消息转换”职责层，统一把业务异常、网络异常、空异常转换成 UI 可消费的文案。
- 定义一个“接口聚合入口”职责层，集中声明所有业务接口方法，确保页面层和控制器层不直接拼 path、method、query、body。
- 定义一个“签名工具”职责层，负责稳定、纯函数式地产生签名。
- 定义一个“加密工具”职责层，负责特定上报接口所需的对称加解密。
- 定义一个“设备公参提供器”职责层，负责异步收集版本号、设备信息、会话标识、时间戳等公参。

## 3. 底层 HTTP 客户端任务

- 初始化底层 HTTP 客户端时设置连接超时和响应超时。
- 默认请求内容类型应与现有模块一致，普通表单请求优先使用 `application/x-www-form-urlencoded` 风格。
- 允许 `validateStatus` 放宽到底层先不过滤业务状态，统一交由上层响应处理。
- 提供获取底层 HTTP 实例的能力，便于后续代理或调试扩展。
- 提供代理设置能力，支持指定代理主机、端口，以及可选跳过证书校验。

## 4. 三类请求行为一致性任务

### 4.1 GET 请求

- 先构建“公参 + 业务参数 + 签名 + 随机串”的完整查询参数。
- 使用统一基础域名拼接相对路径。
- 将完整参数放入查询串发送。
- 收到响应后进入统一响应处理流程。

### 4.2 POST 请求

- 公参、签名、随机串只放查询串。
- 业务参数放请求体。
- 不允许把 POST 的业务参数错误混入签名后的公参查询串，除非项目协议明确要求。
- 收到响应后进入统一响应处理流程。

### 4.3 文件上传请求

- 公参、签名、随机串放查询串。
- 请求体使用 `multipart/form-data`。
- 文件字段名必须允许调用方指定，不能写死为唯一值。
- 业务参数与文件共同写入表单体。
- 文件路径无效时，调用侧可根据具体接口要求回退为普通 POST。

## 5. 统一响应壳任务

- 统一响应模型必须包含以下语义字段：
  - 业务状态码
  - 业务提示消息
  - 数据载荷
- 成功判定必须支持双成功码语义，而不是只认单一成功码。
- 统一响应模型需要支持从动态 JSON 包装对象中解析。
- 当原始响应不是标准对象结构时，必须降级返回一个“非法响应格式”的错误响应，而不是直接崩溃。

## 6. 统一响应处理任务

- 所有请求都必须经过统一响应处理函数。
- 统一响应处理函数先把原始响应解析为统一响应壳。
- 若命中特定“登录失效业务码”，必须触发统一登录失效处理逻辑。
- 登录失效处理完成后，需要抛出业务异常，阻止调用链继续按成功分支执行。
- 若业务状态码不是成功码，必须抛出业务异常，错误文案优先使用服务端返回消息。
- 成功时返回统一响应壳，交由上层解析业务数据。

## 7. 登录失效处理任务

- 登录失效处理必须具备并发幂等保护，避免多个请求同时触发多次跳转和多次清理。
- 登录失效时至少要支持：
  - 清理本地登录态
  - 跳转登录页
  - 输出可追踪日志
- 若项目后续增加全局登出广播或埋点，应在同一统一入口扩展，而不是散落在页面层。

## 8. 公参提供与注入任务

- 支持三种公参来源优先级：
  - 异步提供器
  - 同步提供器
  - 静态公参
- 异步提供器需要能够在应用启动后动态获取设备参数和会话参数。
- 公参内容至少预留以下语义：
  - 应用版本
  - 设备名称
  - 设备唯一标识
  - 系统版本
  - 渠道标识
  - 会话标识
  - 广告标识或等价标识
  - 时间戳
- 时间戳必须由运行时动态生成，不允许写死。

## 9. 公参键名映射任务

- 支持“逻辑字段名 -> 协议字段名”的统一映射。
- 映射逻辑必须集中管理，不能分散在各个接口方法里手工替换。
- 未配置映射的字段保持原 key 透传。
- 由于此部分可能随项目变化，应保留为可配置占位，不强制写入当前项目的具体映射表。

## 10. 签名流程任务

- 签名输入只基于映射后的公参集合，再额外加入“当前请求路径”字段。
- 签名前必须先按 key 做稳定字典序排序。
- 排序后按照 `key + value` 连续拼接成单一字符串。
- 使用 HMAC-SHA256 进行摘要。
- 输出必须是小写十六进制字符串。
- 签名密钥应抽成配置，不允许散落在业务代码各处。
- 签名完成后写回查询参数中。
- 除签名外，还需为每次请求追加一个固定长度的纯数字随机串字段。

## 11. 动态基础域名初始化任务

- 启动时先探测默认 API 域名是否可用。
- 若默认域名可用，直接使用并结束初始化。
- 若默认域名不可用，再请求远端配置地址获取备用域名。
- 远端配置内容可能是明文 JSON，也可能是 Base64 包裹后的 JSON；两种都要兼容。
- 解析成功后至少支持更新：
  - API 基础域名
  - H5 基础域名
- 若远端配置或解析失败，不允许导致应用启动崩溃。
- 若项目需要，可预留“域名切换成功后的日志上报”扩展点。

## 12. 加密工具任务

- 提供对称加密与解密能力。
- 算法、模式、密钥、偏移量必须集中配置。
- 至少支持某些上报接口在调用前先将原始 JSON 字符串加密，再以单字段形式提交。
- 若调用方未提供明文或密文，需要有明确默认行为，不能抛出不可控空值异常。

## 13. 接口聚合层任务

- 所有业务接口必须集中声明，不允许页面层直接写 URL 路径。
- 每个接口方法必须明确：
  - 请求方法
  - 相对路径
  - 参数进入 query、body、multipart 的位置
  - 是否需要补充随机业务字段
  - 是否需要加密后上传
  - 是否需要根据本地文件存在性切换请求分支
- 接口聚合层应返回统一响应壳，不直接在这一层做页面级提示。

## 14. 业务随机字段任务

- 需要支持部分接口额外注入业务侧随机数字串。
- 业务随机字段与公参随机字段是两个概念，不能混为一个统一常量。
- 随机串默认要求：
  - 固定长度
  - 纯数字
  - 每次调用实时生成

## 15. 文件上传分支任务

- 对“可带文件也可不带文件”的接口，必须支持双分支：
  - 本地文件有效时走上传分支
  - 本地文件缺失或为空时走普通表单分支
- 文件存在性判断必须发生在接口聚合层或更高一层可控入口，避免底层无意义构造无效文件对象。

## 16. 统一异常模型任务

- 定义业务异常类型，至少持有一个可读消息字段。
- 网络错误消息转换工具需要覆盖：
  - 空异常
  - 业务异常
  - 底层 HTTP 异常
  - 兜底字符串化异常
- UI 层只消费统一错误消息，不直接依赖底层异常类型细节。

## 17. 页面与控制器接入约束

- 页面层或控制器层只调用接口聚合方法。
- 页面层或控制器层负责把统一响应壳中的业务数据载荷解析成具体业务模型。
- 页面层或控制器层捕获异常时，只通过统一错误消息转换工具取文案。
- 不允许页面层复制登录失效判断、成功码判断、签名逻辑。

## 18. 模型解析任务

- 业务数据模型解析与统一响应壳分层处理。
- 统一响应壳只负责拿到业务载荷，不直接承担所有业务模型解析。
- 上层业务模型需要兼容动态 JSON 节点读取。
- 某些字段若可能为空或类型不稳定，需要通过包装器做防御性解析。

## 19. 可配置项占位清单

- 默认 API 基础域名
- 默认 H5 基础域名
- 远端配置地址
- 公参逻辑字段列表
- 公参键名映射表
- 签名附加的路径字段名
- 签名结果字段名
- 公参随机串字段名
- 签名密钥
- 加密密钥
- 加密偏移量
- 登录失效业务码
- 成功业务码集合

## 20. 严谨性验收清单

- GET、POST、上传三种请求的 query/body/form-data 摆放方式与现有协议一致。
- 成功码判定支持双成功码，而不是单一 `200` 语义。
- 登录失效能被统一捕获，且并发下只执行一次清理与跳转。
- 签名顺序、摘要算法、输出格式完全固定。
- 公参支持异步注入，而不是写死在启动常量里。
- 域名初始化失败不会导致应用崩溃。
- 非标准响应能兜底成业务错误，而不是直接解析异常。
- 业务层不会重复实现签名、公参映射、错误文案转换。
- 文件上传支持字段名定制和文件缺失时的降级分支。
- 加密上报接口支持“传入已加密值”和“传入原始值后内部补加密”两种模式中的目标模式，至少保持与当前项目等价。

## 21. 建议的交付顺序

1. 先完成统一响应壳、异常模型、错误转换工具。
2. 再完成请求配置管理，包括公参来源、公参映射、签名、随机串、域名初始化。
3. 再完成底层请求客户端和三类请求行为。
4. 再完成接口聚合层。
5. 再完成设备公参提供器与启动接入。
6. 最后补齐页面层接入和验证。

## 22. AI 执行约束

- 不要为了“更现代”而把当前协议擅自改造成拦截器全自动黑盒，除非能证明行为完全等价。
- 不要把 POST 的业务体参与签名公参混签，除非项目协议明示如此。
- 不要只实现 happy path，必须保留非法响应、登录失效、域名探测失败、文件不存在等分支。
- 不要省略远端域名探测与 Base64 兼容逻辑。
- 不要把可变的协议字段硬编码成不可替换常量。
- 不要让页面直接感知混淆字段名、签名密钥、加密细节。
