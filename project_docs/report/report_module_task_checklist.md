# Report 模块详细任务清单

本文档用于指导 AI 在任意新项目结构下重建一个与当前项目 report 模块功能和严谨程度完全一致的实现。本文档故意不指定文件名，允许 AI 自由命名，但不允许削弱生命周期时机控制、权限处理、原生数据采集、去重逻辑、缓存回退、第三方 SDK 初始化或上报链路完整性。

## 1. 总目标

- 搭建一个统一的 report 模块，覆盖应用启动时上报、应用恢复时权限刷新、登录后补充上报、位置上报、Google market 上报、风险埋点上报、设备信息上报、推送 token 上报、人脸识别结果上报。
- 输出结果必须与现有模块保持行为等价，而不是仅保留同名方法。
- 可变信息如接口字段名、第三方 SDK token 字段名、缓存键名、具体上报地址可以抽成配置或占位，但处理顺序、去重策略、容错方式、生命周期接入点必须保留。

## 2. 模块职责拆分任务

- 定义一个“report 编排管理器”职责层，统一承接所有 report 入口和生命周期触发。
- 定义一个“payload 构造辅助层”，负责将业务入参、设备信息、定位信息拼装成协议所需结构。
- 定义一个“位置数据模型”职责层，负责原生定位结果解析、缓存结构转换、上报结构转换。
- 定义一个“原生桥接依赖层”，负责获取 IDFA、IDFV、电量、网络、Wi-Fi、定位、推送 token、权限状态等信息。
- 定义一个“缓存依赖层”，负责保存登录时间、首次启动状态、Adjust 初始化状态、位置缓存等。
- 定义一个“第三方归因 SDK 接入层”，负责按需初始化 Adjust 或等价归因 SDK。
- 定义一个“接口调用层依赖”，负责实际发起各类上报请求。

## 3. 生命周期接入任务

### 3.1 应用启动

- 提供一个启动入口方法，只允许真正执行一次，必须带幂等保护。
- 启动时需要完成以下动作：
  - 标记应用已打开并判断是否为首次启动
  - 若首次启动，注册 IDFA 权限变化监听
  - 异步请求通知权限
  - 异步上报原生定位信息
  - 非首次启动时异步上报 Google market
  - 启动推送 token 监听
  - 异步尝试上传 push token

### 3.2 应用回前台

- 提供应用恢复入口方法。
- 应用恢复时只处理“跟踪权限重新请求和 Google market 刷新判断”，不要在这里重复触发所有启动逻辑。
- 恢复时必须具备并发保护，避免重复弹权限或重复串行任务。

### 3.3 登录成功后

- 提供登录成功入口方法。
- 登录成功后至少需要：
  - 记录最新登录时间戳
  - 异步补充原生定位上报
  - 异步上传 push token

## 4. 启动与恢复的权限节流任务

- 通知权限请求与 IDFA 权限请求必须分开处理。
- 启动阶段的权限请求要等待应用处于 resumed 且至少过一帧后再执行，避免过早弹窗。
- 权限请求前后需要延时节流，避免连续系统弹窗冲突。
- 恢复阶段的跟踪权限请求也要等待 resumed 和下一帧。
- 启动阶段和恢复阶段都要有独立的“正在请求”状态，防止并发重入。

## 5. 首次启动 IDFA 监听任务

- 首次启动时，如果需要等待用户对 ATT 权限做出选择，应注册原生权限变化监听。
- 监听状态时要忽略：
  - 空字符串
  - `not_supported`
  - `not_determined`
- 一旦收到最终态：
  - 停止等待首次启动权限结果
  - 标记跟踪市场上报已处理
  - 触发 Google market 上报
  - 取消订阅监听

## 6. Google market 上报任务

- Google market 上报必须通过原生桥接拿到 `idfv` 和 `idfa`。
- 使用 `idfv|idfa` 组成去重签名。
- 当签名为空、已上报过、或当前正在上报时，必须直接跳过。
- 上报成功后要从返回结果中提取第三方 SDK token，并尝试触发 SDK 初始化。
- 上报失败只允许记日志，不能影响主流程。

## 7. 第三方 SDK 初始化任务

- 需要有单独的初始化方法，按需初始化第三方归因 SDK。
- 初始化前要检查：
  - token 非空
  - 本地缓存未标记为已初始化
- 初始化成功后，需要将“已初始化”状态写入项目现有缓存体系。
- 若 SDK 支持 session success / failure 回调，应接入成功和失败日志，并在成功回调里补记已初始化状态。
- 初始化异常不能中断业务主流程。

## 8. 位置获取与位置上报任务

- 提供从原生获取当前位置的方法。
- 同一时刻多处调用位置获取时，必须复用同一个 pending future，避免并发重复拉取原生定位。
- 原生定位成功后要写入本地缓存，供后续埋点或设备信息上报兜底使用。
- 原生定位失败时，只记日志并返回空，不抛出中断主流程的异常。

### 8.1 位置上报触发规则

- 位置上报前必须先校验登录会话存在。
- 获取定位后，如果定位权限不是“未决定”状态，应额外异步触发设备信息上报。
- 若拿不到定位，允许跳过位置上报，但不能阻止设备信息的时机判断逻辑。
- 若拿到定位，则调用位置上报接口。

## 9. 位置缓存回退任务

- 需要支持将定位数据缓存为项目约定结构。
- 风险埋点和设备信息上报在实时定位失败时，必须尝试从缓存恢复位置数据。
- 若缓存中坐标和核心地址信息都为空，应视为无效缓存。

## 10. 风险埋点上报任务

- 提供一个“直接埋点上报”入口，接收已经拼好的字段集合。
- 提供一个“带位置补全的埋点上报”入口，负责自动补齐以下内容：
  - 开始时间
  - 结束时间
  - `idfv`
  - `idfa`
  - 实时位置或缓存位置
- 结束时间必须按秒级时间戳生成。
- 位置获取要带超时兜底，超时后改用缓存位置。
- 埋点字段构造必须集中在 payload helper 中，不能散落在页面层。

## 11. 设备信息上报任务

- 设备信息上报前必须先校验登录会话存在。
- 设备信息采集必须覆盖以下来源：
  - `device_info_plus`
  - `package_info_plus`
  - 原生桥接返回的电量、标识符、elapsed、uptime、代理状态、VPN、越狱、模拟器、语言、运营商、网络类型、时区、CPU 核数、设备名、屏幕尺寸、Wi-Fi、存储信息
  - 本地缓存中的登录时间和位置缓存
- payload 构造完成后必须转 JSON，再做对称加密，再调用上报接口。
- 设备信息上报失败只允许记日志，不能影响其他 report 分支。

## 12. 设备信息 payload 结构任务

- payload 构造必须集中在辅助层。
- 需要明确区分“boot uptime”与“elapsed”两个概念：
  - 原 uptime 字段继续保留给旧字段
  - 新 elapsed 字段单独映射到新的协议字段
- 不允许为了补 elapsed 而覆盖旧 uptime 的语义。
- Wi-Fi 信息需要同时生成：
  - 当前连接对象
  - 网络列表数组
  - 网络数量
- 屏幕宽高应来自项目现有屏幕适配或等价统一来源，而不是页面临时读取。

## 13. Push token 上报任务

- 提供 push token 上传入口。
- 上传流程应先尝试直接读取当前 token。
- 若当前 token 为空，再等待原生 push token 事件流，且要设置超时。
- 若最终仍为空，则静默结束。
- token 上报必须具备去重和并发保护：
  - token 为空时跳过
  - 与最近一次成功上报相同时跳过
  - 与当前正在上报 token 相同时跳过
- 需要常驻监听 push token 变化，一旦有新的有效 token 自动尝试上报。

## 14. 人脸识别结果上报任务

- 提供专门的人脸识别结果上报入口。
- 该入口至少接收：
  - 活体 `id`
  - 请求 `id`
  - 结果码
  - 结果文本
- 上报失败只允许记日志，不影响人脸主流程。

## 15. 通用字符串归一化任务

- report 模块内部应有统一的值转字符串方法。
- 所有上报字段在进入 payload 前需要做空值兼容和 trim 规整。
- 不要在每个调用点分别手写 `toString().trim()`。

## 16. 原生桥接依赖约束任务

- 原生桥接方法需要对未实现情况提供稳定默认值，而不是抛异常。
- 默认值策略至少覆盖：
  - 空字符串
  - `0`
  - 空对象
  - `null`
- 某些字段允许使用 Flutter 侧回退值：
  - 语言可回退系统 locale
  - 时区可回退 `DateTime.now().timeZoneName`
  - CPU 核数可回退平台可获取的近似值

## 17. 缓存依赖约束任务

- report 模块必须复用项目现有缓存机制，不另起一套状态存储。
- 需要明确维护以下状态：
  - 是否首次启动过
  - 登录时间戳
  - Adjust 是否已初始化
  - 最近缓存的位置数据
- 清理登录态时，位置缓存等会话相关数据要一并清理。

## 18. 并发与幂等任务

- 启动入口必须只生效一次。
- Google market 上报必须避免重入。
- 启动权限请求与恢复权限请求必须分别避免重入。
- push token 上报必须避免同 token 并发重复提交。
- 原生定位拉取必须复用 pending future。
- 首次启动 IDFA 监听订阅必须确保只注册一次，并在完成后正确释放。

## 19. 错误处理任务

- 所有 report 分支默认采用“失败记录日志，不中断主业务”的策略。
- 日志内容应走统一错误消息转换工具，避免直接打印复杂异常对象。
- 不允许因为某一个 report 接口失败阻断启动流程、登录流程、定位流程或人脸流程。

## 20. 可配置项占位清单

- 第三方归因 SDK 类型与初始化参数
- 归因 token 返回字段名
- 各 report 接口对应的方法名和参数映射
- 风险埋点字段名映射
- 设备信息 payload 协议字段名映射
- Wi-Fi、存储、电量、位置缓存结构字段名
- 权限请求延迟时间
- push token 等待超时时间
- 实时定位等待超时时间

## 21. 严谨性验收清单

- 启动入口只能触发一次，且能区分首次启动与非首次启动。
- 首次启动监听 IDFA 权限变化后，能在最终态到来时触发 market 上报并释放监听。
- Google market 上报具备签名去重和并发保护。
- Adjust 或等价 SDK 初始化不会重复执行，且成功后会写入现有缓存状态。
- 位置获取并发时不会重复请求原生。
- 定位失败时，风险埋点和设备信息上报可以回退到缓存位置。
- 设备信息 payload 中 `elapsed` 与 `uptime` 是两个独立字段，不会相互覆盖。
- push token 读取、等待、监听、去重、自动补报链路完整。
- 所有 report 分支失败都只记日志，不阻断主流程。
- 登录成功后能刷新登录时间并补发必要上报。

## 22. 建议的交付顺序

1. 先完成位置模型、payload helper、通用字符串归一化。
2. 再完成 report 编排管理器的生命周期入口和状态字段。
3. 再接入缓存依赖和原生桥接依赖。
4. 再完成 Google market、设备信息、位置、push token、风险埋点、人脸结果等具体上报分支。
5. 最后接入第三方 SDK 初始化和权限请求时序控制。

## 23. AI 执行约束

- 不要把 report 模块简化成一个“纯接口调用集合”；它本质上是时序编排器。
- 不要只保留上报方法而丢掉生命周期入口、去重、监听、缓存回退。
- 不要把 `elapsed` 和 `uptime` 合并成一个字段。
- 不要新建独立的本地存储体系替代项目现有缓存实现。
- 不要把原生桥接未实现视为异常中断，必须提供稳定回退值。
- 不要在页面层、控制器层散落拼装 report payload 的逻辑。
